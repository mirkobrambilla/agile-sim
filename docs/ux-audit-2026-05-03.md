# UX Audit - 2026-05-03

Scope: browser review of the current local web app across run picker, scenarios, scenario editor, new-run flow, and runner views.

Audit goals:
- Element visibility and readability
- Aesthetic alignment with the defined "early-2000s sim + vintage chat shell" direction
- Navigation consistency and path predictability
- Control semantics (links vs buttons vs status pills)

## Findings (prioritized)

### 1) Inconsistent control semantics across the app
- Severity: High
- Pages: Global
- Evidence:
  - Mixed styling makes links, buttons, and status pills look similar in multiple surfaces.
  - Some informational chips visually compete with actionable controls.
- User impact:
  - Users cannot predict what is clickable without trial-and-error.
- Recommendation:
  - Standardize semantics globally:
    - Links: navigation only
    - Buttons: actions only
    - Chips/pills: status/tags only (non-interactive)
  - Add stronger visual affordances for each category.

### 2) Home run picker has weak visual hierarchy
- Severity: High
- Pages: `/`
- Evidence:
  - Card text blocks have similar weight; outcome and run type are easy to miss.
  - Scannability is low when many runs appear.
- User impact:
  - Hard to quickly find the right run or understand status at a glance.
- Recommendation:
  - Increase typographic contrast (title vs metadata).
  - Use clearer outcome coding (pass/fail/read-only).
  - Add stronger spacing and grouping.

### 3) Scenario editor app-rail labels are cryptic
- Severity: High
- Pages: `/scenarios/{slug}/edit`
- Evidence:
  - App-rail uses terse initials (`HM`, `SC`, `NR`) without inline meaning.
- User impact:
  - New users cannot confidently navigate editor-level destinations.
- Recommendation:
  - Replace with explicit labels or icons + tooltips.
  - Keep naming consistent with other app areas ("Home", "Scenarios", "New run").

### 4) Channel list interaction affordance is subtle
- Severity: Medium
- Pages: Runner left rail
- Evidence:
  - Channel rows are visually close to static section text.
- User impact:
  - Discoverability of channel navigation suffers.
- Recommendation:
  - Increase clickable-row contrast and default affordance.
  - Improve active/hover states to clearly signal interactivity.

### 5) Back-navigation language is inconsistent
- Severity: Medium
- Pages: Runner, scenario view, scenario editor
- Evidence:
  - Variants like "All runs", "Back to scenarios", and arrow-based labels are used inconsistently.
- User impact:
  - Wayfinding feels less predictable across pages.
- Recommendation:
  - Adopt one pattern and placement for "back" controls.
  - Prefer consistent arrow+destination labels.

### 6) Vitals rail readability needs stronger contrast
- Severity: Medium
- Pages: Runner right rail
- Evidence:
  - Small labels/values and low-contrast chart accents reduce quick legibility.
- User impact:
  - Monitoring character state during runs is slower and more error-prone.
- Recommendation:
  - Increase value/label size and chart contrast.
  - Strengthen panel boundaries for each character block.

### 7) Scenario card action hierarchy is ambiguous
- Severity: Medium
- Pages: `/scenarios`
- Evidence:
  - "Inspect" and "New live run" appear with similar weight, without a clear primary action model.
- User impact:
  - Users hesitate or take inconsistent paths.
- Recommendation:
  - Define and encode a primary action visually.
  - Keep secondary actions visibly secondary (outline/ghost).

### 8) Scenario list cards hide decision-making context
- Severity: Medium
- Pages: `/scenarios`
- Evidence:
  - Cards emphasize slug/counts but provide little scenario intent/context.
- User impact:
  - Requires extra clicks to determine which scenario to use.
- Recommendation:
  - Add short description preview and key tags to cards.

### 9) Turn navigation lacks progress context
- Severity: Medium
- Pages: Runner top bar
- Evidence:
  - Turn controls emphasize step navigation but not overall progress.
- User impact:
  - Users lose orientation in run length/progression.
- Recommendation:
  - Show explicit progress context such as "Turn X of Y" with optional compact progress indicator.

### 10) Read-only replay status can be missed
- Severity: Medium
- Pages: Runner on finished runs
- Evidence:
  - Read-only messaging is low-emphasis relative to main action area.
- User impact:
  - Users attempt actions and then infer behavior from failure/non-response.
- Recommendation:
  - Add a prominent replay-mode banner in top context region.

### 11) Dev panel trigger/content mapping is spatially unclear
- Severity: Low
- Pages: Runner
- Evidence:
  - Trigger location and panel content location feel disconnected.
- User impact:
  - Added cognitive load for internal/dev usage.
- Recommendation:
  - Align trigger and panel placement, or use an explicit overlay model.

### 12) Character representation treatment is inconsistent
- Severity: Low
- Pages: Runner channel/roster/vitals surfaces
- Evidence:
  - Avatar/icon treatment differs by view in ways that feel unintentional.
- User impact:
  - Slight identity-recognition friction and reduced visual cohesion.
- Recommendation:
  - Define avatar size/style variants and apply consistently.

### 13) No clear unread indicators for DM channels
- Severity: Low
- Pages: Runner channel rail
- Evidence:
  - DMs appear uniformly regardless of recent activity.
- User impact:
  - Easy to miss important private updates.
- Recommendation:
  - Add unread badge or activity dot state.

### 14) Some UI tokens read as modern rather than retro-sim
- Severity: Low
- Pages: Global
- Evidence:
  - Rounded modern pills/chips conflict with stated early-2000s simulation vibe.
- User impact:
  - Aesthetic promise is diluted.
- Recommendation:
  - Shift to more boxy/angular tag treatments and retro HUD-like accents where appropriate.

### 15) Timeline presentation is too raw for non-dev users
- Severity: Low
- Pages: Runner timeline
- Evidence:
  - Event display leans toward raw/debug structure rather than narrative timeline comprehension.
- User impact:
  - Harder for non-technical users to extract run story quickly.
- Recommendation:
  - Provide a visual timeline layer; keep raw payload behind a dev toggle.

### 16) Right rail metrics do not prioritize the run-level story
- Severity: High
- Pages: Runner right rail
- Evidence:
  - Current surface emphasizes `work_done` and per-user cards.
  - No combined sentiment trend graph is shown as a primary signal.
- User impact:
  - Users cannot quickly answer "Is team sentiment improving or deteriorating overall?"
  - Coaching decisions are harder because the signal is fragmented.
- Recommendation:
  - Make a combined sentiment trend graph the primary chart in the right rail.
  - Keep per-user cards as drill-down detail rather than top-level focus.

### 17) Summary/report artifacts are underused in runner context
- Severity: High
- Pages: Runner summary/right rail
- Evidence:
  - Key run insights from `summary.json` and `report.md` are not surfaced prominently in the main runner flow.
  - No clear in-place fallback affordance when report output is missing.
- User impact:
  - Users miss high-value conclusions that already exist on disk.
  - Report generation workflow is unclear when artifacts are absent.
- Recommendation:
  - Surface selected `summary.json` outcomes directly in the summary panel (goal result, notable deltas, costs, selected metrics).
  - Render `report.md` highlights inline with a clear "Open full report" action.
  - If report is missing, show a visible "Generate report" button with deterministic empty-state copy.

### 18) Timeline should be a vertical event log with typed visual language
- Severity: High
- Pages: Runner timeline
- Evidence:
  - Current timeline does not visually separate event classes enough for fast reading.
  - Coach actions and notable metric changes are not strongly highlighted.
- User impact:
  - Users cannot rapidly scan causality between interventions and outcomes.
- Recommendation:
  - Redesign timeline as a vertical chronological log.
  - Use explicit event-type styles (coach action, agent post, metric shift, rule edit, system event).
  - Add visual emphasis for coach-originated events and threshold-crossing metric changes.

### 19) Channel selectors need message volume and unread state
- Severity: Medium
- Pages: Runner left rail
- Evidence:
  - Channel list currently lacks persistent message counts and obvious unread indicators.
- User impact:
  - Users cannot prioritize where to look next in busy runs.
- Recommendation:
  - Show per-channel message count badges.
  - Add unread indicators (dot/count) with clear active/read state transitions.

### 20) View switches should persist in URL for back-button behavior
- Severity: High
- Pages: Runner view navigation
- Evidence:
  - View transitions are primarily dynamic swaps; URL state does not always reflect current view.
- User impact:
  - Browser back/forward behavior feels broken or surprising.
  - Deep-linking specific runner views is harder.
- Recommendation:
  - Encode current view (and selected channel/filters/turn where relevant) in URL query/state.
  - Update history on view change so back/forward restores prior view state.

### 21) Roster cards should carry richer decision-making context
- Severity: Medium
- Pages: Runner roster
- Evidence:
  - Roster cards prioritize identity but provide limited actionable context.
- User impact:
  - Users must click through repeatedly to compare who needs attention.
- Recommendation:
  - Add key compact details on each card (top vitals, active work, recent trend, risk marker).
  - Keep card density readable while improving at-a-glance triage value.

### 22) Inspector content density exceeds drawer ergonomics
- Severity: Medium
- Pages: Runner inspector
- Evidence:
  - Inspector payload can exceed comfortable drawer width/flow for deeper objects.
- User impact:
  - Dense information is harder to scan and compare in a narrow side panel.
- Recommendation:
  - Add dedicated full-page inspect/profile routes for rich entities.
  - Keep drawer as quick inspect; provide "Open full page" as standard affordance.

### 23) Channel view should anchor to newest messages by default
- Severity: High
- Pages: Runner channel view
- Evidence:
  - Initial viewport behavior does not always prioritize the latest messages.
- User impact:
  - Users land away from the most relevant recent conversation.
- Recommendation:
  - Default message pane scroll position to bottom on channel open/switch.
  - Preserve expected behavior when user manually scrolls upward.

### 24) Channel layout should use sticky header + full-height scroll body
- Severity: High
- Pages: Runner center column, channel view
- Evidence:
  - Scrollable region and header do not always maximize usable chat space.
- User impact:
  - Less efficient conversation review and composition flow.
- Recommendation:
  - Make channel header/controls sticky at top of center column.
  - Let message list own the remaining column height as the primary scroll container.
  - Keep composer consistently positioned and accessible.

## Top 5 action points

1. Normalize control semantics and styling rules app-wide (link vs button vs chip).
2. Redesign run-picker card hierarchy for fast scanning and clearer status.
3. Replace editor rail shorthand labels with explicit navigation language.
4. Rework runner information architecture: combined sentiment graph + summary/report integration.
5. Make runner interactions URL-stateful and chat-first (vertical timeline, unread/counts, bottom-anchored message pane).
